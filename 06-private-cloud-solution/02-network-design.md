# 从0到企业级私有云：网络设计方案（核心基础篇）

## 一、为什么网络设计是私有云核心基础

在企业级私有云建设中，计算资源决定性能上限，存储资源决定数据可靠性，而 **网络设计决定整个平台是否稳定可扩展**。

大量 Kubernetes 集群问题，本质都来自网络规划不合理，例如：

- 节点 IP 混乱，后期无法扩容
- Pod 网段冲突
- Service 网段与办公网冲突
- 跨节点通信异常
- DNS 解析不稳定
- 南北流量入口混乱
- 存储流量抢占业务带宽

因此，网络设计必须在集群部署前完成规划。

---

# 二、企业级私有云网络总体分层

建议采用逻辑分层设计：

```text
┌────────────────────────────┐
│ 外部访问层                 │
│ 用户 / 浏览器 / API Client │
└────────────────────────────┘

┌────────────────────────────┐
│ 接入层                     │
│ LB / Ingress / WAF         │
└────────────────────────────┘

┌────────────────────────────┐
│ 集群业务层                 │
│ Service / Pod Network      │
└────────────────────────────┘

┌────────────────────────────┐
│ 节点管理层                 │
│ SSH / kubelet / etcd       │
└────────────────────────────┘

┌────────────────────────────┐
│ 存储网络层（可选独立）     │
│ NFS / Ceph / Backup        │
└────────────────────────────┘
````

---

# 三、推荐网络划分（生产思路）

建议最少拆分以下网络：

| 网络类型       | 作用            | 是否建议独立 |
| ---------- | ------------- | ------ |
| 管理网        | SSH、API、节点管理  | 是      |
| 业务网        | 用户访问业务服务      | 是      |
| Pod 网络     | 容器互联          | 系统内部   |
| Service 网络 | 集群虚拟服务        | 系统内部   |
| 存储网        | NFS/Ceph 数据流量 | 强烈建议   |
| 监控网（可选）    | 指标采集          | 可选     |

# 六、Kubernetes 网络参数规划（极重要）

## Pod 网段

```text
10.244.0.0/16
```

或：

```text
172.16.0.0/16
```

要求：

* 不与公司办公网冲突
* 不与宿主机网段冲突
* 后期可扩容

---

## Service 网段

```text
10.96.0.0/12
```

默认常用。

---

## DNS Service IP

```text
10.96.0.10
```

CoreDNS 使用。

# 七、CNI 网络插件选型建议

## 方案一：Calico（推荐多数场景）

优点：

* 稳定成熟
* 文档多
* 支持 NetworkPolicy
* 支持 BGP

适合：

* 中小企业
* 通用生产环境

---

## 方案二：Cilium（进阶）

优点：

* eBPF 高性能
* 可观测性强
* 安全策略强大

适合：

* 高并发
* 云原生高级场景

---

# 八、南北流量设计（外部访问）

用户访问流程：

```text
Internet / 内网用户
        ↓
负载均衡器
        ↓
Ingress Controller
        ↓
Service
        ↓
Pod
```

建议部署：

* NGINX Ingress
* HAProxy
* Keepalived + Nginx
* MetalLB（裸金属环境）

---

# 九、东西流量设计（集群内部）

集群内部流量：

```text
Pod A → Pod B
Service → Pod
Node → Node
```

要求：

* 延迟低
* MTU 正确
* 防火墙放行 VXLAN/BGP 端口
* DNS 稳定

# 十一、DNS 设计建议

企业私有云必须保证 DNS 稳定。

建议：

```text
业务域名：
jenkins.company.local
grafana.company.local
argocd.company.local
harbor.company.local
```

内部解析方式：

* 企业 DNS
* dnsmasq
* 路由器静态解析（实验环境）

# 十三、常见踩坑总结

## 1. Pod 网段冲突

现象：

```text
Pod 无法通信
```

原因：

公司内网已使用同网段。

---

## 2. MTU 错误

现象：

```text
偶发超时、大包丢失
```

---

## 3. DNS 不稳定

现象：

```text
Pod 间歇性连接失败
```

---

## 4. 节点使用 DHCP

现象：

```text
IP 变化导致集群异常
```

必须固定 IP。

---

# 十四、实验环境最佳实践（你的场景）

推荐：

```text
单网卡 + 单网段
固定IP
Calico
MetalLB
Ingress Nginx
```

简单稳定，适合持续实验。

---

# 十五、企业生产最佳实践

推荐：

```text
双网卡
管理网独立
业务网独立
存储网独立
Calico / Cilium
硬件 LB
双核心交换机
```

---

# 十六、总结

网络是私有云最容易被低估的部分。

很多人把 Kubernetes 装起来了，却败在网络细节：

* 网段规划
* 流量分层
* DNS 稳定
* 高可用入口
* 扩容预留

网络设计做好，平台才能长期稳定演进。
